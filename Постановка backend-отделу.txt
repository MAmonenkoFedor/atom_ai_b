
Построить production-ready backend для ATOM AI Workspace, который:

* поддерживает структуру компании с иерархией,
* даёт проектный и организационный контекст,
* подключает внешние источники,
* обеспечивает AI-чат и summaries,
* масштабируется на компанию ~500 сотрудников,
* и главное — стабильно обслуживает существующий frontend-контракт. 

---

## 1. С чего начинаем

Начинаем **не с коннекторов и не с AI**, а с **фундамента**.

### Этап старта

Сначала backend-команда должна поднять 4 базовых слоя:

1. **Platform foundation**

   * Django monolith skeleton
   * DRF
   * PostgreSQL
   * Redis
   * Celery
   * MinIO/S3
   * settings: local / stage / prod
   * health checks
   * logging / Sentry / metrics

2. **Identity + Organization**

   * user
   * organization
   * org hierarchy
   * roles
   * memberships
   * department/team structure

3. **API contract layer**

   * `/api/v1`
   * OpenAPI schema
   * единый error format
   * единый pagination format
   * `snake_case` responses
   * согласованные query params: `q`, `status`, `sort`, `page`, `page_size`

4. **Workspace core**

   * projects
   * chats
   * messages
   * ai_run skeleton

Порядок именно такой, потому что фронт уже построен вокруг стабильного API-фасада, normalizers и hooks, а также ожидает Django/DRF-совместимую форму данных. Это делает backend API-контракт задачей первого порядка, а не “потом допишем”.

### Внутри каждого app

```text id="61606"
apps/<domain>/
  admin/

    serializers.py

    filters.py
  models/
```

### Правило структуры

* `models` — только данные и связи
* `serializers/api` — только транспортный слой
* `services` — бизнес-логика
* `selectors` — чтение/выборки
* `tasks` — фоновые процессы
* `permissions.py` — объектные и доменные проверки доступа

Это нужно, чтобы не свалиться в типичный хаос “всё в views.py”.

## 4. Что backend должен гарантировать фронту

Это нужно прямо зафиксировать как правило команды.

### Backend contract rules

1. Все endpoint’ы идут через `/api/v1`
2. Все поля на сервере — `snake_case`
3. Все списки имеют одну пагинацию
4. Все list-endpoints поддерживают `q`, `status`, `sort`, где это применимо
5. Все ошибки возвращаются в одном формате
6. Все долгие операции идут через async job/run
7. AI-ответы обязаны возвращать `run_id`, `status`, `citations`, `usage`, `provider`, `model`
8. Backend не ломает поля молча без согласования с frontend

Это полностью совпадает с тем, как у вас уже устроено расширение сущностей и endpoint’ов на фронте: `liveClient.ts`, `normalizers.ts`, контракт `ApiClient`, фасад `lib/api`, hooks и отсутствие прямых HTTP-вызовов из UI.

## 9. Правила работы backend-отдела с frontend-командой

Это надо зафиксировать как обязательный процесс.

### Любой новый endpoint считается готовым только если:

* есть serializer
* есть OpenAPI schema
* согласована response shape
* известны query params
* есть error cases
* фронт может добавить normalizer
* фронт может добавить метод в `liveClient.ts`
* фронт может пробросить это через `lib/api`

Это напрямую вытекает из вашего playbook: новые сущности и endpoint’ы должны проходить через `services/api`, normalizers, контракт `ApiClient`, фасад `lib/api` и hooks, а UI не должен делать сырые HTTP-запросы. 

### Общие правила

* backend не меняет поля без согласования
* backend не делает “временные костыльные” endpoint’ы мимо `/api/v1`
* frontend не ходит в неописанные endpoint’ы
* долгие операции только через jobs/runs
* для AI всегда есть trace и status

## 10. Что отдать backend-отделу прямо сейчас

Вот короткая формулировка задачи, которую можно отправить команде.

Задача backend-отделу:

Нужно начать разработку production-ready backend для ATOM AI Workspace на Django + DRF + PostgreSQL + Redis + Celery.

Первый этап — не коннекторы и не сложный AI, а фундамент системы:

1. поднять backend skeleton и окружения;
2. реализовать identity, organization и hierarchy;
3. зафиксировать API contract под текущий frontend;
4. реализовать projects, chats, messages и ai_run skeleton.

Backend должен сразу работать под существующий frontend-подход:

* `/api/v1`
* `snake_case`
* единый pagination format
* единый error format
* query params `q/status/sort/page/page_size`
* OpenAPI как источник истины
* async jobs/runs для долгих операций
* SSE/stream для AI-чатов

Структура backend должна быть доменной:
core, identity, organizations, orgstructure, projects, chats, ai, connectors, sync, documents, mail, issues, knowledge, llm_gateway, usage_billing, activity, audit, files.

Первый deliverable:

* stage environment
* auth + me
* organization hierarchy
* employees/org units endpoints
* projects endpoints
* chats/messages endpoints
* ai_run skeleton
* базовый OpenAPI contract

После этого переходим к llm gateway, затем к connectors/sync, затем к retrieval и citations.

Если нужно, следующим сообщением я превращу это в **Jira-структуру: Epic → Story → Task**.
